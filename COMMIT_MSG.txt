feat: Add project-specific workflow filtering with safe fallback

Implements optional per-project workflow filtering to solve the issue of
multiple workflows being triggered when only one workflow should run for
a specific project type.

## Problem

When multiple workflows exist in a repository (e.g., gerrit-packer-verify.yaml,
gerrit-jjb-verify.yaml, gerrit-docs-verify.yaml), ALL workflows were triggered
on every change, regardless of project type. This caused unnecessary CI runs
and resource waste.

## Solution

Added a new `job_filter` parameter to the workflow filtering system that allows
project-specific workflow selection through configuration, with automatic
fallback to triggering all workflows if not configured (100% backward compatible).

### Implementation Details

1. **Core Functionality (github.py)**
   - Added `job_filter` Optional[str] parameter to `filter_workflows()`
   - Implements 3-level filtering: "gerrit" + event_type + job_filter
   - Maintains backward compatibility when job_filter is None
   - Case-insensitive matching for all filter levels

2. **Configuration Support (config.py)**
   - Added `get_project_workflow_filter(project, event_type)` function
   - Reads optional per-project configuration from INI file
   - Returns filter string (e.g., "packer") or None if not configured
   - Format: [project "name"] with verify_filter/merge_filter options

3. **Integration (helpers.py)**
   - Updated `find_and_dispatch()` to retrieve project-specific filters
   - Passes job_filter to both regular and required workflow filtering
   - Preserves exact_match functionality alongside job_filter

### Configuration Example

```ini
[project "releng/builder"]
verify_filter = packer
merge_filter = packer

[project "ci-management"]
verify_filter = jjb
merge_filter = jjb
```

### Behavior

**With configuration:**
- Project "releng/builder" + patchset-created → triggers ONLY gerrit-packer-verify.yaml
- Project "ci-management" + patchset-created → triggers ONLY gerrit-jjb-verify.yaml

**Without configuration (backward compatible):**
- Any project + patchset-created → triggers ALL gerrit-*-verify.yaml workflows

## Testing

Added comprehensive test coverage with 4 new test functions:

1. **test_filter_workflows_maven_patterns**
   - Tests Maven-specific workflows: gerrit-verify-maven.yaml,
     gerrit-verify-maven-mri.yaml
   - Validates filter "maven" matches all maven workflows
   - Validates filter "maven-mri" matches only MRI-specific workflow

2. **test_filter_workflows_complex_patterns**
   - Tests 11 real-world workflows across multiple tech stacks
   - Covers Packer, Maven, Python, JJB naming patterns
   - Validates both naming orders: gerrit-<job>-verify and gerrit-verify-<job>
   - Tests compound patterns: maven-clm, python-tox, maven-sonar

3. **test_filter_workflows_case_insensitive**
   - Validates case-insensitive matching: "maven" == "MAVEN" == "Maven"
   - Ensures consistent behavior across all case variations

4. **test_filter_workflows_required_with_job_filter**
   - Tests job_filter integration with required workflows
   - Validates correct filtering of organization-level workflows

## Key Features

✅ **100% Backward Compatible** - Projects without config trigger all workflows
✅ **No Workflow Changes** - Keep existing workflow names unchanged
✅ **Zero Performance Impact** - No SSH queries, config-based only
✅ **Safe Fallback** - Missing config = trigger everything (never miss workflows)
✅ **Flexible Patterns** - Supports any naming pattern (gerrit-job-verify, gerrit-verify-job)
✅ **Case Insensitive** - Filter matching works with any case
✅ **Well Tested** - Comprehensive test coverage with real-world scenarios
✅ **Gradual Rollout** - Deploy without config, add project configs incrementally

## Migration Path

1. Deploy updated code (no config changes needed)
2. Existing behavior: all workflows trigger (no change)
3. Add config for pilot projects
4. Gradually expand configuration to remaining projects
